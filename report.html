Seven tags reviewed. Two high-severity issues, one consent gap, and a handful of housekeeping items. Your conversion pixel is firing twice on checkout — fix that first.

01 · Findings

What CAT spotted

Duplicate conversion pixel on /checkout/success
Tag Google Ads — Purchase fires twice on the success page — once via All Pages trigger, once via Purchase event. Revenue is double-counted for ~40% of sessions.
tag: Google Ads — Purchase · trigger: All Pages + Purchase
High
GA4 session_start missing user_id
Logged-in users aren't stitching across devices. The session_start event does not include user_id, but your User ID variable is populated.
tag: GA4 Configuration · variable: dlv.user_id (present)
High
Consent mode defaults not set before GTM loads
The default consent command runs inside GTM instead of before. A race condition means some users send hits with ad_storage=granted by accident.
container: default · cmp: OneTrust
Medium
4 paused tags left in container
Legacy tags are paused but still shipping in every container publish. Safe to delete.
tags: fb_legacy, pinterest_v1, tiktok_test, hotjar_old
Low
Enhanced ecommerce dataLayer is clean
All required parameters present on view_item, add_to_cart, and purchase.
events: view_item, add_to_cart, purchase
OK
02 · Recommendations

What to do

  1. Remove the All Pages trigger from Google Ads — Purchase

    Keep only the Purchase event trigger. Validate in Preview mode on /checkout/success before publishing.

  2. Add user_id to the GA4 Configuration tag

    Map {{dlv.user_id}} to the user_id field. This enables cross-device identity stitching in GA4.

  3. Move consent defaults into the <head>

    Run gtag('consent', 'default', ...) inline, before the GTM snippet. OneTrust's template supports a pre-GTM hook.

  4. Delete the four paused legacy tags

    Archive the container version first so rollbacks stay easy.
